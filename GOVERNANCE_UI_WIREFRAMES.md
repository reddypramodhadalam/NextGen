# Human-In-The-Loop Governance — UI Wireframes & Flow

> **Principle:** _AI can ASSIST, but can NEVER AUTHOR or APPROVE. Final accountability must always be human._

This document shows the exact disclaimer + approval flow as implemented in the codebase. ASCII wireframes are paired with the React component / route that renders them.

## 9. Backend Enforcement Map

| User Action | Frontend Component | Backend Middleware | DB Effect |
|---|---|---|---|
| Generate AI tests | `<AiDisclaimerBanner variant="generator">` | `POST /api/test-cases` → auto-stamp `reviewStatus=DRAFT` + audit `AI_TEST_CASE_GENERATED` | Row created, status=DRAFT |
| Save AI tests | `<HumanReviewGate>` opens after save | `POST /api/governance/reviews/bulk` | review_records row + audit `REVIEW_APPROVED` + status=APPROVED |
| Edit approved test | (inline editor) | `PATCH /api/test-cases/:id` → un-approves if content changed | status reverts to DRAFT, hash cleared |
| Click "Run Suite" | (error toast 409) | `POST /api/executions` → `requireApprovedTestCases` | Audit `EXECUTION_BLOCKED_NO_REVIEW` |
| Toggle auto-heal in healer | Switch disabled in VALIDATED | `POST /api/healer/heal` → `blockAutoApplyAiFix` | 403 if `autoHeal=true` + CRITICAL audit |
| Click "Apply Fix" | `<HumanReviewGate>` opens | `POST /api/healer/apply` → requires `approvalId` from review | Audit `AI_HEAL_APPLIED` |
| Upload screenshot to AQM | `<EvidenceReviewDialog>` | `POST /api/governance/evidence/:id/upload` → `requireEvidenceReview` | 409 if not all 3 attestations |
| Switch VALIDATED ↔ NON | `<RegulatoryModeCard>` AlertDialog | `PUT /api/governance/system-type` | platform_settings + audit `SYSTEM_TYPE_CHANGED` |

## 10. Non-Bypassable Properties

1. **No "skip review" feature flag.** Mode is determined by `platform_settings.system_type` — there is no per-request override.
2. **No admin bypass.** Admin users hit the same middleware as everyone else.
3. **Edits invalidate approval.** `PATCH /api/test-cases/:id` clears `contentHash` and reverts status to DRAFT if any content-changing field is modified. Hash mismatch is also detected at execution time by `isReviewStillValid()`.
4. **Multi-approver support.** `SETTING_KEY_MIN_APPROVERS` controls how many distinct approvers must sign per `content_hash` before status flips to APPROVED.
5. **E-signature must match account.** `HumanReviewGate` enforces this on the client; backend re-validates against session user.
6. **Audit log is append-only.** No DELETE / UPDATE endpoints. Signatures are SHA-256 over `eventType|resourceType|resourceId|actorId|timestamp|payload`. Verifiable via `GET /api/governance/audit/verify/:id`.
7. **Bypass attempts are CRITICAL.** Calling `/api/healer/apply` without `approvalId` in VALIDATED mode logs a CRITICAL `REVIEW_BYPASS_ATTEMPTED` event with full request payload (tokens redacted).
